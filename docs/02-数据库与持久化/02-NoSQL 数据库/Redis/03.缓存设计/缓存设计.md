# 缓存设计

## 什么是缓存击穿？

缓存击穿是指在高并发系统中，一个非常热门的键（hot key）在缓存过期失效时，大量请求同时尝试从数据库中读取该数据，导致数据库瞬间承受巨大压力，甚至可能崩溃的现象。这类似于“击穿”了缓存层，直接打到后端存储。

### 为什么会发生？

- 缓存通常设置过期时间，以避免数据永久不一致
- 当热门数据过期时，如果没有及时续期或预加载，大量并发请求会同时miss缓存，转而查询数据库。
- 数据库查询较慢，容易形成瓶颈。

### 典型场景

- 电商平台的热门商品库存：在秒杀活动中，库存数据缓存过期时，大量用户查询导致数据库负载飙升。

### 如何避免？

#### 互斥锁

使用分布式锁（如Redis的SETNX），只允许一个请求去数据库加载数据，其他请求等待或返回旧值。如果结合Redis，可以用Lua脚本或Redisson实现锁机制来缓解。

- 原理：当缓存miss时，使用分布式锁（如Redis锁）确保只有一个请求去DB加载数据，其他请求等待锁释放后从缓存读取。防止并发穿透。

- 实现步骤：

  1. 尝试获取锁（e.g.，Redis SETNX）
  2. 获取成功：从DB加载，写入缓存，释放锁
  3. 获取失败：等待重试或返回默认值/旧值
  4. 设置锁超时等待避免死锁

- 代码示例

  ```java
  // Java+Redisson分布式锁库，推荐比纯Jedis可靠
  
  import org.redisson.Redisson;
  import org.redisson.api.RLock;
  
  public String getWithMutex(String key) {
      RLock lock = redisson.getLock("lock:" + key);
      String value = jedis.get(key);
      if (value != null) return value; // 缓存命中
  
      try {
          if (lock.tryLock(10, 60, TimeUnit.SECONDS)) { // 尝试获取锁，等待10s，持有60s
              value = queryFromDatabase(key); // 从DB查询
              if (value != null) {
                  jedis.setex(key, 300, value); // 写入缓存
              }
              return value;
          } else {
              // 锁获取失败：等待或返回null/旧值
              Thread.sleep(100); // 简单重试
              return getWithMutex(key); // 递归重试（生产限次）
          }
      } finally {
          lock.unlock();
      }
  }

  ```java
  // 使用Lettuce作为Redis客户端
  import io.lettuce.core.RedisClient;
  import io.lettuce.core.api.StatefulRedisConnection;
  import io.lettuce.core.api.sync.RedisCommands;
  
  public class CacheBreakdownExample {
      private static final String LOCK_KEY = "lock:product:123"; // 锁key
      private static final String CACHE_KEY = "product:123"; // 缓存key
  
      public String getProductInfo(String productId) {
          RedisClient redisClient = RedisClient.create("redis://localhost:6379");
          StatefulRedisConnection<String, String> connection = redisClient.connect();
          RedisCommands<String, String> commands = connection.sync();
  
          // 尝试从缓存获取
          String cachedData = commands.get(CACHE_KEY);
          if (cachedData != null) {
              return cachedData;
          }
  
          // 缓存miss，尝试获取锁（使用SETNX，过期时间防止死锁）
          boolean locked = commands.setnx(LOCK_KEY, "locked"); // SET if Not eXists
          if (locked) {
              commands.expire(LOCK_KEY, 10); // 锁10秒过期
              try {
                  // 查询数据库（模拟）
                  String dbData = queryFromDatabase(productId);
                  // 更新缓存
                  commands.set(CACHE_KEY, dbData, "EX", 300); // 缓存5分钟
                  return dbData;
              } finally {
                  // 释放锁
                  commands.del(LOCK_KEY);
              }
          } else {
              // 未获取锁，等待重试或返回默认值
              Thread.sleep(50); // 短暂等待
              return getProductInfo(productId); // 递归重试（实际中可限次）
          }
      }
  
      private String queryFromDatabase(String productId) {
          // 模拟数据库查询
          return "Product info from DB";
      }
  }
  ```

-  优缺点：

  | 方面     | 优点                 | 缺点                           |
  | -------- | -------------------- | ------------------------------ |
  | 实现难度 | 中高，需要分布式锁   | 高并发下锁竞争可能导致延迟     |
  | 性能影响 | 轻微，锁粒度细       | 锁超时需调优，避免雪崩         |
  | 适用场景 | 通用，尤其突发热门键 | 不适合超高QPS（用Lua脚本优化） |



#### 逻辑过期

- 原理：不设置真是的TTL（过期时间），而是在value汇总嵌入一个逻辑过期时间字段。当获取缓存时，检查是否“逻辑过期”。如果过期，只有一个线程去异步更新数据库，其他线程返回旧值。

- 优点：避免锁竞争，适合读多写少的场景

- 缺点：需要额外字段管理，可能有短暂的数据不一致

- 代码示例

  ```java
  // 使用JSON序列化value
  
  import com.fasterxml.jackson.databind.ObjectMapper;
  import io.lettuce.core.RedisClient;
  // ... 其他import
  
  public class LogicalExpirationExample {
      private static final String CACHE_KEY = "product:123";
  
      static class CacheValue {
          String data;
          long expireTime; // 逻辑过期时间（Unix timestamp）
      }
  
      public String getProductInfo(String productId) throws Exception {
          RedisClient redisClient = RedisClient.create("redis://localhost:6379");
          // ... 获取commands
  
          String json = commands.get(CACHE_KEY);
          if (json == null) {
              // 缓存miss，直接查询DB并设置
              return rebuildCache(commands, productId);
          }
  
          ObjectMapper mapper = new ObjectMapper();
          CacheValue value = mapper.readValue(json, CacheValue.class);
  
          // 检查逻辑过期
          if (System.currentTimeMillis() > value.expireTime) {
              // 逻辑过期，异步重建（或用锁控制一个线程重建）
              new Thread(() -> {
                  try {
                      rebuildCache(commands, productId);
                  } catch (Exception e) {
                      // 处理异常
                  }
              }).start();
              // 返回旧数据
              return value.data;
          }
  
          return value.data;
      }
  
      private String rebuildCache(RedisCommands<String, String> commands, String productId) throws Exception {
          String dbData = queryFromDatabase(productId);
          CacheValue newValue = new CacheValue();
          newValue.data = dbData;
          newValue.expireTime = System.currentTimeMillis() + 300000; // 逻辑过期5分钟后
  
          ObjectMapper mapper = new ObjectMapper();
          commands.set(CACHE_KEY, mapper.writeValueAsString(newValue)); // 无TTL
          return dbData;
      }
  
      // queryFromDatabase 同上
  }
  ```

#### 永不过期

#### 监控Redis的命中率

Redis的缓存命中率是评估缓存效率的关键指标，通常计算公式为：

`命中率=keyspace_hits/(keyspace_hits+keyspace_misses)*100%`

其中keyspace_hits是缓存命中的次数、keyspace_misses是缓存未命中的次数。

理想情况下，命中率保持在90%以上，如果低于70%，可能需要优化缓存策略（例如增加缓存容量、调整过期时间）



**如何监控**

1. 使用Redis内置命令（INFO）：

   - Redis提供INFO命令来获取服务器统计信息，包括hits和misses

   - 在Java中，可以使用Redis客户端（如Lettuce或Jedis）定期执行`INFO status`，解析输出计算命中率

   - 示例Java代码

     ```java
     // 使用Lettuce
     
     import io.lettuce.core.RedisClient;
     import io.lettuce.core.api.StatefulRedisConnection;
     import io.lettuce.core.api.sync.RedisCommands;
     import java.util.Map;
     
     public class RedisMonitor {
         public double getCacheHitRatio() {
             RedisClient redisClient = RedisClient.create("redis://localhost:6379");
             StatefulRedisConnection<String, String> connection = redisClient.connect();
             RedisCommands<String, String> commands = connection.sync();
     
             // 执行INFO stats
             String info = commands.info("stats");
             Map<String, String> stats = parseInfo(info); // 自定义解析方法
     
             long hits = Long.parseLong(stats.getOrDefault("keyspace_hits", "0"));
             long misses = Long.parseLong(stats.getOrDefault("keyspace_misses", "0"));
             long total = hits + misses;
     
             return total > 0 ? (double) hits / total * 100 : 0.0;
         }
     
         private Map<String, String> parseInfo(String info) {
             // 简单解析INFO输出为Map
             Map<String, String> map = new java.util.HashMap<>();
             for (String line : info.split("\n")) {
                 if (line.contains(":")) {
                     String[] parts = line.split(":");
                     map.put(parts[0].trim(), parts[1].trim());
                 }
             }
             return map;
         }
     }
     ```

   - 使用：在服务中，每分钟调用一次`getCacheHitRatio()`，记录到日志或监控系统。如果是生产环境，结合Scheduled任务自动化监控

2. 集成监控工具：

   - Prometheus + Redis Exporter：Redis Exporter是一个sidecar，将Redis metrics暴露给Prometheus。安装后，在Prometheus中查询redis_keyspace_hits_total和redis_keyspace_misses_total，计算命中率。
   - **RedisInsight**：Redis官方GUI工具，支持实时监控hits/misses，支持警报。
   - **Grafana**：结合Prometheus可视化命中率图表。
   - **云服务**：如果使用AWS ElastiCache或阿里云Redis，提供内置监控面板，直接查看命中率。

3. 报警设置：

   - 当命中率低于阈值时，出发警报（如通过Prometheus Alertmanager发送邮件/Slack）




#### 监控数据库QPS

数据库QPS（Queries Per Second）指每秒执行的查询次数，常用于评估数据库负载。假设后端是MySQL（常见于Java+Redis架构），QPS可以通过MySQL的status变量计算：

`QPS=(Questions-previous_Questions)/time_interval`

其中，Questions是累计查询数



**如何监控**

1. 使用数据库内置命令：

   - MySQL：执行SHOW GLOBAL STATUS LIKE 'Questions'; 获取累计查询数，定期计算增量得出QPS。

   - 在Java中，使用JDBC或Spring Data JPA连接数据库，执行查询。

   - 示例Java代码（使用HikariCP连接池）：

     ```java
     import com.zaxxer.hikari.HikariConfig;
     import com.zaxxer.hikari.HikariDataSource;
     import java.sql.Connection;
     import java.sql.ResultSet;
     import java.sql.Statement;
     
     public class DatabaseMonitor {
         private HikariDataSource dataSource;
         private long previousQuestions = 0;
         private long previousTime = System.currentTimeMillis();
     
         public DatabaseMonitor() {
             HikariConfig config = new HikariConfig();
             config.setJdbcUrl("jdbc:mysql://localhost:3306/yourdb");
             config.setUsername("root");
             config.setPassword("password");
             dataSource = new HikariDataSource(config);
         }
     
         public double getQPS() throws Exception {
             try (Connection conn = dataSource.getConnection();
                  Statement stmt = conn.createStatement();
                  ResultSet rs = stmt.executeQuery("SHOW GLOBAL STATUS LIKE 'Questions'")) {
     
                 if (rs.next()) {
                     long currentQuestions = Long.parseLong(rs.getString("Value"));
                     long currentTime = System.currentTimeMillis();
                     long deltaQuestions = currentQuestions - previousQuestions;
                     long deltaTime = (currentTime - previousTime) / 1000; // 秒
     
                     previousQuestions = currentQuestions;
                     previousTime = currentTime;
     
                     return deltaTime > 0 ? (double) deltaQuestions / deltaTime : 0.0;
                 }
             }
             return 0.0;
         }
     }
     ```

   - 使用：类似Redis，定时调用getQPS()，监控峰值QPS是否超过数据库容量（例如，MySQL单机QPS上限约1-10万，视硬件）。

2. 集成监控工具：

   - Prometheus + MySQL Exporter：Exporter暴露MySQL metrics，包括mysql_global_status_questions，计算QPS。
   - Percona Monitoring and Management (PMM)：免费工具，支持MySQL QPS监控和警报。
   - New Relic 或 Datadog：云监控服务，集成Java应用，自动采集数据库QPS。
   - Spring Boot Actuator：如果用Spring，启用/actuator/metrics端点，结合Micrometer暴露JDBC metrics。

3. 警报与优化：

   - 设置QPS阈值警报（e.g., >5000 QPS触发）。
   - 优化：如果QPS高且Redis命中率低，可能是缓存失效导致；增加读从库、优化SQL。

**综合建议**

- 统一监控平台：使用Prometheus + Grafana监控Redis和数据库，创建仪表盘显示命中率和QPS曲线。
- 日志与追踪：结合ELK（Elasticsearch + Logstash + Kibana）或Sentry记录异常。
- 性能测试：用JMeter模拟高并发，观察指标变化。
